## Problem

The Income Tax Calculator is hardcoded to FY 2025-26 (AY 2026-27).
Today is 2026-05-05 — FY 2026-27 has begun, Budget 2026 has been delivered,
and visitors who want to plan their FY 2026-27 tax liability cannot.
Meanwhile, taxpayers are still actively filing returns for FY 2025-26
(ITR filing window April–July 2026), so removing the older year would
also break a legitimate use case.

CEO-review reframe: The literal ask is "add FY 2026-27" — but the deeper
problem is twofold: (1) annual maintenance burden each February when
slabs change; (2) a silent senior-citizen gap in the Old Regime path
since v1. The FY-keyed config refactor solves both.

## User

- **Primary**: Indian salaried individuals (incl. Srinidhi himself)
  comparing Old vs New Regime for FY 2026-27 planning.
- **Secondary**: Indian senior citizens (60–80) and super senior
  citizens (80+) who currently get the wrong number under Old Regime.

## Solution: Approach A — FY-keyed config refactor

Refactor `IncomeTaxCalculator.js` around a `TAX_CONFIG[fy]` object that
holds every FY-dependent value. Add two UI controls:

1. **FY pill toggle** at top of card (P1) — `FY 2025-26 (AY 26-27)` |
   `FY 2026-27 (AY 27-28)`. Default = current FY based on date.
   Caption underneath: "FY = Financial Year, Apr–Mar" (P3).
2. **Age category pills** inside Old Regime panel (P2) —
   `General (<60)` | `Senior (60–80)` | `Super Senior (80+)`.
   Progressive disclosure: only renders when regime === 'old'.

`computeTaxForRegime()` and `computeSurchargeWithRelief()` become
FY-parameterized; PDF generator likewise dynamic.

Mobile UX: results panel auto-scrolls into view on FY/regime change (P4).

## Architecture

```
IncomeTaxCalculator.js  (UI, ~1100 lines)
   │  state: fy, regime, ageCategory, income, deductions, results
   │  effects: recompute on inputs; scrollIntoView on FY/regime; reconcileDeductions on FY
   │
   ├─→ tax-config.js  (NEW)
   │    TAX_CONFIG['2025-26' | '2026-27']
   │      newRegime { slabs, standardDeduction, rebate }
   │      oldRegime { slabs:{general,senior,superSenior}, standardDeduction, rebate }
   │      surcharge { brackets, oldMax, newMax }
   │      cessRate
   │      deductionSections [...]
   │    getCurrentFY(date) → '2026-27'  (Apr 1 cutover, fallback latest)
   │
   └─→ Tax engine (kept inline — pure functions)
        computeTaxForRegime(fy, regime, ageCategory, gross, deductions)
        computeSurchargeWithRelief(fy, regime, taxable, baseTax, slabs)
        computeBaseTaxFromSlabs(slabs, taxable)
```

## Alternatives Considered

- **Per-FY computation modules** — high duplication, drift risk; rejected.
- **Inline `if (fy === '2026-27')` branches** — won't scale; rejected.
- **Extract engine to its own file** — over-engineering for v1; defer.

## Edge Cases (15 catalogued, 3 needed work)

1. **Default FY by current date** — Apr 1 cutover; fallback to latest if user-FY > configured FY → `getCurrentFY()` helper [IT-3]
2. Senior + New Regime — age ignored (engine reads regime first)
3. FY toggle preserves income/regime/age/deductions
4. **Deduction sections may differ across FYs** → `reconcileDeductions()` on FY change [IT-2]
5. Section caps applied at input time only (FY switch doesn't shrink user values)
6. 87A rebate FY-specific (read from config)
7. PDF filename includes selected FY
8. PDF compares both regimes for SELECTED FY only (no cross-FY)
9. Surcharge brackets/rates FY-specific
10. New-regime ₹12L marginal relief (Budget 2025) — research must confirm threshold for FY 2026-27
11. Old Regime "phasing out" hint if research flags it
12. Existing tests must keep passing (IT-1 verifies before IT-2 ships)
13. localStorage persistence — out of scope, future task
14. URL-encoded shareable state — out of scope, future task
15. **Worked-examples snapshot acceptance** — research doc's 3 worked examples must reproduce to ±₹1 [IT-9]
